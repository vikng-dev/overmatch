# NaN/Placeholder rollback bug — lightyear 0.28.0 example & book research

Source: lightyear repo clone at tag 0.28.0 (`scratchpad/lightyear-src`). All paths relative to the clone root. All line numbers from actual reads of this clone.

## Q1: `examples/deterministic_replication` — how entities get `DeterministicPredicted`, and what they carry at that moment

There are **three** decoration sites; in every one of them the entity has real (never placeholder) `Position` at decoration time, and physics components arrive **in the same `spawn`/`insert` call** as `DeterministicPredicted` or before it — never after.

1. **Ball (shared, both peers)** — `examples/deterministic_replication/src/shared.rs:150-176` (`spawn_ball`): one `commands.spawn((...))` containing `Position::default()` (explicit, shared.rs:157), `PhysicsBundle::ball()` (Collider + ColliderDensity + RigidBody + Restitution, protocol.rs:17-33), and `DeterministicPredicted { skip_despawn: true, ..default() }` (shared.rs:161-164). Position exists with a real value from the entity's very first tick.

2. **Server player** — `examples/deterministic_replication/src/server.rs:134-148` (`handle_connected`): one `commands.spawn((...))` with `player_bundle(remote_id.0)` — which contains an explicit `Position::from(Vec2::new(-50.0, y))` (shared.rs:178-187) — plus `DeterministicPredicted { skip_despawn: true, ..default() }` (server.rs:144-147). The in-code comment at server.rs:139-143 is load-bearing: *"`skip_despawn: true` — the player is spawned by a non-deterministic event... Rollback must not despawn the entity, and `enable_rollback_after: N` prevents restoring history for ticks before the entity existed on this peer."*

3. **Client player activation (the closest analog to our async bind)** — `examples/deterministic_replication/src/client.rs:72-110` (`activate_physics_when_bundle_lands`, an `On<CatchUpSnapshotReady>` observer, registered client.rs:29). It inserts `PhysicsBundle::player()` + `DeterministicPredicted { skip_despawn: true }` onto an already-existing replicated entity — i.e., physics IS added after spawn here. But two things differ from our pattern: (a) the entity already carries authoritative replicated `Position`/`Rotation` from the catch-up snapshot **before** the collider is inserted (the observer only fires after `CatchUpSnapshotReady`, i.e. "after all gated state arrives", client.rs:66-71; the server replicates `Position` etc. per protocol.rs:115-136); (b) the collider bundle and `DeterministicPredicted` land **atomically in one insert** (client.rs:99-109), so there is never a tick where a physics component exists on the entity without the rollback-grace marker.

**Do decorated entities carry avian `Position`/`Rotation`?** Yes — `Position` always explicitly (shared.rs:157, shared.rs:182), and both `Position` and `Rotation` are registered for `local_rollback` with confirmed writes, custom hashes, interpolation and correction (protocol.rs:115-129). `Rotation` is never explicitly inserted in spawn bundles in this example — it comes in via avian2d required components (UNCERTAIN whether avian2d 0.x's `Rotation` default is a finite identity vs. a placeholder here; avian source is not in this clone. In avian2d the example never crashes on it, and in the client activation path `Rotation` is replicated state before the collider lands).

**How the example avoids restoring placeholder history — the actual mechanism (lightyear core, not example code):**
- `DeterministicPredicted` has a built-in grace field: `enable_rollback_after: u8`, default **20 ticks** — `crates/replication/prediction/src/rollback.rs:255-275`.
- Its `on_add` hook records `(decoration_tick + enable_rollback_after, entity)` into `PredictionManager.deterministic_skip_despawn` — rollback.rs:277-300 (note rollback.rs:282-291: the hook **silently no-ops if the `PredictionResource`/`PredictionManager` doesn't exist yet** — the protection is not applied in that case).
- When any rollback fires, the tail of `check_rollback` inserts **`DisableRollback`** on every skip-despawn entity whose protection tick is still in the future (`rollback_tick < protection_tick`), and removes it from entities whose window has passed — rollback.rs:767-812 (insert at :799, removal at :807).
- `RollbackSystems::Check → RemoveDisable → Prepare → Rollback → EndRollback` are `.chain()`ed (rollback.rs:121-132), so the `DisableRollback` command applies before `prepare_rollback` runs.
- `prepare_rollback<C>` queries `Without<DisableRollback>` (rollback.rs:883-891), so protected entities are **entirely skipped from restore**. During rollback the whole entity is also hidden from all queries via `DisabledDuringRollback` (rollback.rs:303-311, 1103-1109) and un-hidden at rollback end (rollback.rs:1145-1147).
- Restore semantics: `prepare_rollback` restores `predicted_history.get_state(rollback_tick)` (rollback.rs:932-936), and `get_state` is **at-or-before** semantics (`crates/core/core/src/history_buffer.rs:160-168`). So if junk is ever recorded at an entity's first tick, any rollback landing at-or-after that tick (and before the next history entry) restores the junk — unless the `DisableRollback` grace window covers it. If no history entry exists at-or-before the rollback tick, the component is left untouched (rollback.rs:964-974), so an *empty* history is safe; a *junk-seeded* history is not.

Also relevant: the example adds `FrameInterpolate<Position/Rotation>` via an `On<Add, DeterministicPredicted>` observer (shared.rs:96-116) specifically so post-rollback `Position` isn't clobbered by stale `Transform` at FixedUpdate start (shared.rs:29-37) — a reminder that with `PhysicsTransformPlugin` disabled (shared.rs:46-55) the examples are explicit about who owns transform↔position sync.

## Q2: Does any example/demo add colliders/physics to a predicted entity AFTER spawn?

**No example uses async asset-driven collider binding.** `grep` for `ColliderConstructor`, `ColliderConstructorHierarchy`, `SceneRoot`, `.glb`, `.gltf`, `GltfAssetLabel` across `examples/` and `demos/` returns **zero hits**. No example loads a rig/scene and binds colliders when the asset resolves. Our ~1s-deferred glb bind has **no reference precedent** in this repo.

What DOES exist is the "blueprint pattern": physics bundles inserted onto a predicted entity **at the moment it becomes Predicted** (same frame or the next, via `Added<Predicted>` queries or `On<Add, ...>` observers), when the entity already carries authoritative replicated `Position` (and `Rotation` where registered):

- `examples/avian_3d_character/src/client.rs:65-79` — `handle_new_character`: `Added<Predicted> + With<CharacterMarker>` → insert `CharacterPhysicsBundle` (Collider/RigidBody/LockedAxes/Friction, **no** Position/Rotation — those are replicated per protocol.rs:88-100). Similar: floors client.rs:112-122 (`Added<Remote>`), blocks client.rs:125-135 (`Added<Predicted>`).
- `examples/avian_physics/src/client.rs:41-51` — `add_ball_physics` (`On<Add, BallMarker>`, if Predicted) and client.rs:86-102 `handle_predicted_spawn` (`On<Add, (PlayerId, Predicted)>` → `PhysicsBundle::player()`).
- `examples/projectiles/src/client.rs:53-61,67-80` — Collider+RigidBody inserted on predicted/interpolated players on add; client.rs:82-100 `handle_deterministic_spawn` inserts `player_bundle(...)` (which itself contains explicit `Position::from_xy` AND `Rotation::default()` alongside the Collider — shared.rs:1935-1954) **plus** `DeterministicPredicted { skip_despawn: true }` in one insert.
- `demos/spaceships/src/client.rs:61-81` — `add_bullet_physics`: `With<BulletMarker>, Without<Collider>` → insert `PhysicsBundle::bullet()` on predicted bullets / sensor collider on interpolated ones.
- `examples/deterministic_replication/src/client.rs:72-110` — the catch-up activation described in Q1 (physics + `DeterministicPredicted` atomic, after authoritative Position/Rotation landed).

Common invariants across every site: (1) the insert happens within one frame of the entity becoming Predicted/replicated — never seconds later mid-gameplay; (2) `Position` (the prediction-registered transform state) is already present with a real value BEFORE or ATOMICALLY WITH the collider; (3) the insert targets the **root** entity only; (4) where `DeterministicPredicted` is involved, it lands in the same `insert`/`spawn` as the physics bundle.

## Q3: `DisableRollback` usage in `avian_3d_character` and `projectiles`; spawn bundle completeness

### avian_3d_character
- `examples/avian_3d_character/src/renderer.rs:181-199` — `disable_projectile_rollback`: query `With<Predicted>, With<ProjectileMarker>, Without<DisableRollback>` → `commands.entity(proj).insert(DisableRollback)`. Registered in `Last` (renderer.rs:48) with the comment *"We disable rollbacks for projectiles after the initial rollbacks which brings them to the predicted timeline"*; setup intent at renderer.rs:29-32: *"we use ReplicateOnce and DisableRollback to stop replicating any packets for these bullets"*.
- **Lifecycle: one-way.** No `.remove::<DisableRollback>()` exists anywhere in `examples/` or `demos/` (grep). Once a projectile has it, it is permanently excluded from rollback restore (`prepare_rollback` filters `Without<DisableRollback>`, `crates/replication/prediction/src/rollback.rs:890`) and hidden during rollback execution via `DisabledDuringRollback` (rollback.rs:1103-1109, removed at rollback end :1145-1147). The only removal of `DisableRollback` in the whole repo is lightyear-internal, for `DeterministicPredicted` skip-despawn entities whose grace window expired (rollback.rs:787, :807).
- Purpose: the projectile is predicted-spawned and needs exactly one initial state sync onto the predicted timeline; after its first predicted frame, its trajectory is fully deterministic locally, so rollback restore would only cause it to "jump" — hence permanent opt-out. The `Last`-schedule timing guarantees at least one full frame (including any rollback that synced it) has completed.
- Spawn completeness: server projectile spawn (`examples/avian_3d_character/src/server.rs:92-109`) includes explicit `*position` AND `Rotation::default()` AND `LinearVelocity` in the bundle. Character spawn (server.rs:176-191) has explicit `Position(Vec3::new(x, 3.0, z))` but **no explicit Rotation** — Rotation arrives via avian required components (UNCERTAIN what initial value avian3d assigns there; avian source not in this clone). Floor/block (server.rs:116-131): explicit `Position::new(...)`, no explicit Rotation, both with colliders at spawn.

### projectiles
All five `DisableRollback` sites are **at spawn time, in the spawn bundle** of client-side visual/local-sim projectile entities (not predicted entities):
- `examples/projectiles/src/shared.rs:1480-1506` hitscan visual (:1502);
- shared.rs:1509-1550 linear bullet (:1545) — with the load-bearing comment at :1542-1544: *"The entity is not predicted, so we want to disable it during rollbacks, otherwise it will start jumping forward on rollbacks"*;
- shared.rs:1553-1602 shotgun pellets (:1596); shared.rs:1604-1648 physics projectile (:1642); shared.rs:1650-1687 homing missile (:1684).
- Every one of those bundles contains explicit `position` AND `rotation` (copied from the shooter's components at fire time) as the first two bundle members (e.g. shared.rs:1531-1532, :1581-1582, :1624-1625, :1668-1669). The physics projectile additionally has `Collider::circle` + `RigidBody::Dynamic` in the same bundle (:1627-1628).
- `player_bundle` (shared.rs:1935-1954): explicit `Position::from_xy(0.0, y)` AND `Rotation::default()` AND `Collider::rectangle` AND `RigidBody::Kinematic` — the complete transform+physics state in one bundle.
- Lifecycle: same as above — inserted once, never removed.

**Pattern summary:** examples use `DisableRollback` in two shapes: (a) permanent opt-out for locally-deterministic predicted projectiles after their first predicted frame (avian_3d_character), (b) at-spawn opt-out for non-predicted visual entities that merely carry `Position`/`Rotation` (prediction-registered types) so rollbacks never touch them (projectiles). Shape (b) is exactly the "cosmetic entity carrying prediction-registered components" hazard — the example's answer is to mark them `DisableRollback` from birth.

## Q4: RayCaster entities and child colliders under predicted bodies — ALL examples

**(a) `RayCaster`: zero occurrences.** grep for `RayCaster`/`ShapeCaster` across `examples/` and `demos/` finds nothing. All raycasting is done through the `SpatialQuery` system param (e.g. `examples/avian_3d_character/src/shared.rs:120,149`; `examples/projectiles/src/shared.rs:327,508`; `examples/fps/src/server.rs:121,180`) or `LagCompensationSpatialQuery` (`examples/fps/src/server.rs:16`; `examples/projectiles/src/server.rs:30`), which adds **no components to any entity**. No example ever has a `RayCaster`-component entity, so no example ever needed to answer whether such an entity carries explicit `Position`/`Rotation`. Our RayCaster-component-on-entity pattern is also off the reference map.

**(b) Child colliders under a predicted rigid body: none exist.** Every `with_children`/`ChildOf` use in `examples/` and `demos/` is cosmetic or UI: bullet visual meshes (`demos/spaceships/src/renderer.rs:455-465`), wall visual meshes (renderer.rs:610-619), entity name labels (`demos/spaceships/src/entity_label.rs:62,72`), UI nodes (`examples/auth/src/client.rs:164`; `examples/common/src/client_renderer.rs:73`, `examples/common/src/server_renderer.rs:66`). A grep for `Collider`/`RigidBody` inside `with_children` blocks returns zero hits. In every example, the collider lives on the **same entity** as the rigid body and the prediction markers. `projectiles`' `BulletOf` (shared.rs:1500,1546) is a custom relationship, not a transform/physics hierarchy. **No example anywhere attaches physics components to child entities of a predicted body.**
