# Release artifacts and repository layout: four directory roles, one staged bundle

The repo separates content by **what consumes it and when**, not by file type. Four top-level roles, each with one job:

- **`assets/`** — *runtime-loaded* content only. Loaded by Bevy's `AssetServer` at runtime, relative to the working directory. Large binaries (`*.glb`, `*.blend`) are Git LFS-tracked (`.gitattributes`). Blender sources (`*.blend`/`*.blend1`) sit beside their exported `*.glb` for authoring convenience but are **pruned from shipped bundles** — only the `.glb` the game actually loads travels with the binary.
- **`build/`** — *build-time inputs*, never loaded at runtime. `build/branding/` holds the master art (sources of truth: `logo_master*.png`); `build/icons/` holds the **generated, committed** icon deliverables (`window_icon.png`, `icon.icns`, `icon.ico`); platform packaging inputs (e.g. `build/macos/` entitlements + `Info.plist`) live here too. Consumed at compile/package time — `branding.rs` bakes `build/icons/window_icon.png` into the binary via `include_bytes!`.
- **`dist/`** — *local packaging output*. Gitignored, ephemeral, regenerated by `scripts/build-linux.sh`. Its staged folder (binary + pruned `assets/`) is **exactly what a player receives** — and is also exactly a Steam depot's content folder, so the same staging serves both the GitHub-Release archive and a future SteamPipe upload.
- **`target/`** — Cargo's build cache. Gitignored, machine-specific, never curated by hand.

The through-line: **the truth committed to git is `src/` + `Cargo.{toml,lock}` + `assets/` + `build/` sources**; everything a user runs is *produced* from a tagged commit and *hosted* elsewhere (GitHub Releases now, Steam later). `dist/` and `target/` are byproducts.

Why this shape, and not the obvious alternatives:

- **`assets/` is runtime-only, on purpose.** Bevy resolves assets from `./assets` at runtime; branding/icons must work *before* an asset path exists (and inside a relocated macOS `.app`), so they are embedded from `build/`, not loaded from `assets/`. Mixing branding into `assets/` would ship source art to players and couple the window icon to the working directory.
- **Icons are generated from the emblem, and committed.** `scripts/gen-icons.sh` derives all platform icons from `build/branding/logo_master_bore.png` (the emblem — *not* the wordmark: text-in-icon is illegible at 16×16). Outputs are committed rather than regenerated in CI so the build needs no image tooling and `include_bytes!` always resolves; re-run the script when the master art changes.
- **One staged bundle, two destinations.** Keeping packaging layout (`dist/`) identical to a Steam depot means the build/package work is reused verbatim when Phase 2 (Steam) lands — only the publish step differs.

## Considered Options

- **Everything under `assets/`** (icons, source art, runtime meshes together). Simplest, but ships source art, breaks icon loading from a relocated `.app`, and blurs "what the player downloads." Rejected.
- **Generate icons in CI / a `build.rs`** instead of committing them. Purer single-source-of-truth, but forces image tooling (ImageMagick) into every build, complicates `include_bytes!`, and slows CI for art that changes rarely. Rejected in favor of commit-the-output + a documented regen script.
- **`cargo-dist` for the whole release pipeline.** Declarative and excellent for CLI tools, but its installer model (curl|sh, Homebrew, MSI) targets tools, not a game with large LFS assets, a signed `.app`, and Steam as the real endpoint. Rejected; we hand-author the workflow.
- **`cargo-packager` for the macOS `.app`.** Considered (declarative), but the sibling `autoquit` project already has a proven local `codesign`/`notarytool`/`stapler` flow, and a Bevy binary → `.app` is a trivial folder layout. We mirror that flow in `scripts/package-macos.sh` (assemble universal `.app` → sign with hardened runtime + Steam entitlements → notarize → staple → `.dmg`) for full control and reuse of a known-good pattern.

## Consequences

- **Release flow:** bump `version` in `Cargo.toml` → commit → `git tag vX.Y.Z` → `git push --tags`. The tag triggers `.github/workflows/release.yml`, which builds Linux/Windows/macOS in a matrix and publishes per-platform archives named `overmatch-<tag>-<target>.{tar.gz,zip}` (+ `.sha256`) to a GitHub Release. Regular pushes to `main` do **not** release.
- **Versioning:** loose semver while `0.x`. A breaking **on-disk format** change (saves/replays) is the trigger to reserve a `MAJOR` bump; artifacts should carry their own embedded format version for the actual compatibility check (app version ≠ format version).
- **Binary size:** `[profile.release]` uses `strip = true` + `lto = "thin"` + `codegen-units = 1` (smaller/faster at the cost of slow, LTO-heavy CI builds — acceptable for an infrequent release cadence).
- **CI must fetch LFS** (`actions/checkout` with `lfs: true`) in the bundling job, or assets ship as broken pointer files.
- **macOS** ships a signed + notarized `.app` (entitlements/`Info.plist` under `build/`), which doubles as the Steam macOS depot. Windows/Linux ship a binary + `assets/` folder (zip/tarball; AppImage optional later).
